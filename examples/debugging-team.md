# 경쟁 가설 디버깅 팀 프롬프트 예제

## 기본 구성: 과학적 토론

```
사용자가 앱에서 메시지 하나 보낸 후 연결이 끊긴다고 보고했어.
5명의 에이전트 teammate를 소환해서 각각 다른 가설을 조사해줘.
서로 가설을 반박하는 과학적 토론처럼 진행하고,
합의된 결론을 findings 문서에 업데이트해줘.
```

### 왜 이 구성이 효과적인가

순차 조사는 **앵커링 효과**에 취약합니다. 첫 번째 가설이 탐색되면 이후 조사가 그 방향으로 편향됩니다. 여러 독립 조사자가 서로의 이론을 적극적으로 반박하면, 살아남는 이론이 실제 근본 원인일 가능성이 높습니다.

## 변형: 3가설 집중

```
API 응답 시간이 느려졌어. 3명으로 원인 조사:
- 가설 A: 데이터베이스 쿼리 문제 (인덱스, N+1)
- 가설 B: 네트워크 레이턴시 (외부 API 호출)
- 가설 C: 메모리 누수 (캐시 관리)
각자 증거를 수집하고, 서로의 가설을 도전해줘.
```

## 변형: 크래시 분석

```
Firebase에서 크래시 리포트가 급증했어. 스택트레이스:
[스택트레이스 붙여넣기]

3명의 teammate로 분석:
- 스택트레이스 파싱 + 심볼 추적
- 최근 코드 변경 이력 추적 (git blame)
- 동일 이슈 재현 조건 탐색
서로 발견 사항을 공유하고 합의해줘.
```

## 팁

- **반드시 "서로 반박하라"고 명시** → 이것이 핵심 메커니즘
- teammate 수는 가설 수에 맞추기 (3-5명)
- Lead에게 "최종 합의가 없으면 가장 유력한 2개 가설을 보고"하도록 지시
